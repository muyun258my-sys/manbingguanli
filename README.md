# 慢病健康管理助理系统

> 基于 Multi-Agent + RAG 架构的慢病管理 AI 助手，支持症状分析、用药咨询与就医导诊，面向高血压、糖尿病、冠心病等慢性病患者及家属。

---

## 项目背景

慢性病患者在日常管理中面临三类高频痛点：不知道当前症状是否需要就医、不确定多种药物能否同时服用、不清楚应该挂哪个科室。传统搜索引擎给出的信息零散且缺乏个性化，本项目通过 Multi-Agent 架构将这三类需求分别交由专属 Agent 处理，结合用户健康档案提供上下文感知的回答。

---

## 系统架构

```
用户输入
    │
    ▼
Orchestrator Agent          ← 意图分类 · 任务路由 · 安全边界判断
    │
    ├──────────────────────────────────┐
    │                                  │
    ▼                                  ▼
症状分析 Agent          用药管理 Agent          导诊 Agent
RAG 检索医学指南        禁忌检查 · 用药提醒      严重度分级 · 科室推荐
    │                       │                       │
    └───────────────────────┴───────────────────────┘
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
        医学知识库      用户健康档案      药品数据库
     指南 PDF · Qdrant  病史·用药·过敏史  成分·禁忌·剂量
              │
              ▼
       DeepSeek LLM（推理 / 生成）
              │
              ▼
     Streamlit 前端 + FastAPI 接口
```

## 核心模块说明

### Orchestrator Agent

系统的任务调度中心，负责：

- **意图分类**：将用户输入分为症状描述、用药咨询、就医导诊、多意图混合、档案管理、高风险输入六类
- **任务路由**：单意图直接分发给对应 Agent；多意图任务拆解后串行或并行调用多个 Agent
- **安全边界判断**：双重检查机制——关键词匹配（"剧烈胸痛"、"昏迷"等）+ LLM 快速分类（`is_emergency: true/false`），任一触发则 short-circuit，跳过所有 Agent 直接输出就医提示
- **结果聚合**：多 Agent 结果存在矛盾时，取更保守的结论（如导诊 Agent 与症状 Agent 结论冲突，优先建议就医）

### 症状分析 Agent

- 接收用户症状描述，结合用户健康档案（已知病史、过敏史）构建查询
- 使用 Hybrid Search（稠密向量 + BM25 稀疏检索）在医学指南知识库中检索
- 通过 BGE-Reranker 对检索结果做二次排序，提升相关性
- 将 Top-K 文档块与用户档案一同注入 Prompt，由 DeepSeek 生成分析结论

### 用药管理 Agent

- 解析药品说明书 PDF，提取成分、适应症、禁忌、用法用量等结构化字段存入数据库
- 处理用药查询时结合结构化查询（SQL）与语义检索（RAG）两种方式
- 支持多药联合禁忌检查：用户提供当前用药清单，Agent 逐一比对禁忌字段
- 输出结构化的用药建议，LLM 负责最终的自然语言组织

### 导诊 Agent

- 基于症状分析结果进行严重度三级分类：
  - 🟢 可自行观察（调整生活方式即可）
  - 🟡 建议近期就诊（1 周内挂号）
  - 🔴 立即就医（当日或急诊）
- 根据症状类型推荐对应科室（内分泌科、心内科、神经内科等）
- 🔴 级别结果会触发 Orchestrator 的界面置顶提示逻辑

## 用户输入类型与路由策略

| 类型 | 示例 | 路由目标 |
|------|------|----------|
| 症状描述 | "最近头晕，血压测了 160/100" | 症状分析 Agent |
| 用药咨询 | "二甲双胍和阿司匹林能一起吃吗" | 用药管理 Agent |
| 就医导诊 | "这个情况需要去医院吗，挂哪个科" | 导诊 Agent |
| 多意图混合 | "血糖高，吃着二甲双胍，要换药吗还是去医院" | 多 Agent 串行 |
| 档案管理 | "我上周查出来有高血压，对青霉素过敏" | 档案写入逻辑 |
| 高风险输入 | "突然胸口剧痛，左臂也麻了" | Short-circuit → 强制就医提示 |

| 云部署 | Railway | 后端服务部署 |
| 数据存储 | SQLite / PostgreSQL | 用户健康档案、药品结构化数据 |

## RAG 优化方案

相比基础 RAG（全字段拼接 + 单一向量检索），本项目采用以下优化：

1. **Metadata 过滤**：按疾病类型（糖尿病/高血压/冠心病）过滤候选文档，缩小检索范围
2. **Hybrid Search**：稠密向量（语义相似度）+ BM25 稀疏检索（关键词匹配）加权融合
3. **BGE-Reranker 重排**：对 Top-20 检索结果做精排，取 Top-5 进入最终 Prompt
4. **Query 改写**：多轮对话中对指代模糊的问题（"那这个药呢"）做上下文补全后再检索

---

## 多轮对话与记忆管理

- 使用 `ConversationBufferWindowMemory` 维持最近 5 轮对话上下文
- 用户健康档案（病史、用药清单、过敏史）作为 System Prompt 固定注入，每轮对话均可感知
- 档案更新实时写入数据库，下一轮对话立即生效

## 安全边界设计

医疗 AI 的核心风险是"过度自信"——系统给出错误建议而用户未就医。本项目通过以下机制控制风险：

- **免责声明**：所有输出在界面显示"本系统仅供参考，不构成医疗建议"
- **高风险 Short-circuit**：双重检测触发后，系统停止生成任何医疗建议，仅显示急救提示
- **严重度置顶**：导诊结果为🔴时，界面强制置顶显示就医建议，不可被其他内容遮挡
- **拒绝诊断**：系统不对疾病做确诊性表述，只输出"可能相关"、"建议咨询医生确认"等措辞

## 快速启动

### 一键启动（推荐）

安装依赖后，用一条命令同时拉起后端（8080）和前端（8501），前端会自动打开浏览器：

```bash
pip install -r requirements.txt
```

- **Windows**：双击 `start.bat`，或在 CMD 中运行 `start.bat`
- **Git Bash / WSL / macOS / Linux**：`bash start.sh`

脚本会先启动后端、等待 3 秒后启动前端。前端侧边栏的"后端地址"已默认指向 `http://127.0.0.1:8080`，无需修改。

> `start.bat` 在前端关闭后，后端窗口（标题 `app-backend`）仍会保留，手动关闭即可；`start.sh` 则在前端退出时自动关闭后端。

如需分别手动启动，见下方步骤。

---

### 手动启动

#### 1. 安装依赖

```bash
pip install -r requirements.txt
```

#### 2. 启动后端（FastAPI）

```bash
python -m uvicorn app.app:app --host 127.0.0.1 --port 8080
```

或直接运行包入口（默认 8080 端口）：

```bash
python -m app
```

启动后访问 `http://127.0.0.1:8080/docs` 可查看交互式 API 文档。

#### 3. 启动前端（Streamlit）— 新开一个终端

```bash
python -m streamlit run frontend/app.py
```

浏览器自动打开 `http://localhost:8501`。若后端不在默认 8080 端口，在前端左侧边栏的"后端地址"输入框中修改。

#### 4. 运行测试

```bash
python -m pytest tests/ -v
```

## 项目亮点总结

1. **Multi-Agent 架构**：Orchestrator + 3 专属 Agent，职责清晰，具备真实业务场景的任务编排逻辑
2. **RAG 全链路优化**：Metadata 过滤 + Hybrid Search + Reranker，可量化对比基础 RAG 的检索质量提升
3. **安全边界设计**：双重高风险检测 + Short-circuit 机制，体现对医疗 AI 产品责任边界的思考
4. **工程化落地**：FastAPI + Streamlit + Railway + Qdrant Cloud，完整的云端部署链路
5. **上下文感知**：健康档案持久化 + 多轮对话记忆，体现对真实用户场景的理解
