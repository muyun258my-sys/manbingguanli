## 1. 架构总览

系统分为六层，自上而下为：表现层、接口层、编排层、能力层、服务层、数据层。
另有一条独立于在线请求链路的**离线数据管道**，负责知识库构建。

```
┌─────────────────────────────────────────────────────────────┐
│  表现层   Streamlit（多轮对话 UI + 健康档案管理）              │
├─────────────────────────────────────────────────────────────┤
│  接口层   FastAPI（/chat · /profile · /health）               │
├─────────────────────────────────────────────────────────────┤
│  编排层   Orchestrator（意图分类 · 安全网关 · 路由 · 聚合）    │
├─────────────────────────────────────────────────────────────┤
│  能力层   症状分析 Agent │ 用药管理 Agent │ 导诊 Agent         │
├─────────────────────────────────────────────────────────────┤
│  服务层   RAG 检索 · LLM · 档案 · 安全检测 · 会话记忆          │
├─────────────────────────────────────────────────────────────┤
│  数据层   Qdrant(向量) · SQLite/PG(结构化) · Memory(会话)      │
└─────────────────────────────────────────────────────────────┘

         ╔══════════════════════════════════════════╗
         ║  离线 Ingestion 管道（系统启动前预跑）      ║
         ║  PDF → 解析 → 切片 → 向量化/抽取 → 入库     ║
         ╚══════════════════════════════════════════╝
```

## 2. 分层职责

### 2.1 表现层（Streamlit）
- 多轮对话窗口，渲染结构化回答（含严重度标识、免责声明）
- 健康档案管理面板：病史、用药清单、过敏史的查看与编辑
- 🔴 高严重度结果的界面置顶/强提示渲染
- 仅做展示与交互，不含业务逻辑，所有数据经接口层获取

### 2.2 接口层（FastAPI）
- 对外暴露 RESTful 接口，做请求校验、会话标识管理、异常包装
- 不承载业务决策，仅将请求转交编排层并回传结果
- 统一注入免责声明与响应包络（envelope）

### 2.3 编排层（Orchestrator）
任务调度中心，由四个子组件串联：
1. **安全网关**：最先执行，命中高风险则 short-circuit
2. **意图分类器**：六类意图判定
3. **路由器**：单意图直发 / 多意图拆解后串并行调度
4. **聚合器**：合并多 Agent 结果，冲突时取最保守结论

### 2.4 能力层（三个子 Agent）
各 Agent 接口契约统一（见 §4），内部实现各异：
- **症状分析 Agent**：RAG 驱动的症状研判
- **用药管理 Agent**：SQL + RAG 混合的用药/禁忌咨询
- **导诊 Agent**：严重度分级 + 科室推荐

### 2.5 服务层（共享能力）
被多个 Agent 复用的无状态/有状态服务：
- RAG 检索服务、LLM 服务、档案服务、安全检测服务、会话记忆服务

### 2.6 数据层
- Qdrant：医学指南向量（按疾病类型 Metadata 分区）
- SQLite/PostgreSQL：用户档案、药品结构化数据
- Memory：会话窗口（最近 5 轮）

## 3. 接口层 API 契约

> 所有响应统一包络：`{ code, message, data, disclaimer }`

### 3.1 `POST /chat`
| 字段 | 方向 | 类型 | 说明 |
|------|------|------|------|
| `session_id` | 入 | string | 会话标识，缺失时由服务端生成 |
| `user_id` | 入 | string | 用户标识，关联健康档案 |
| `message` | 入 | string | 用户自然语言输入 |
| `reply` | 出 | string | 最终自然语言回答 |
| `intent` | 出 | enum | 命中的意图类型 |
| `severity` | 出 | enum\|null | 🟢/🟡/🔴，仅导诊相关时有值 |
| `emergency` | 出 | bool | 是否触发 short-circuit |
| `sources` | 出 | array | RAG 引用的文档来源（标题/出处） |

### 3.2 `GET /profile/{user_id}` · `PUT /profile/{user_id}`
| 字段 | 类型 | 说明 |
|------|------|------|
| `conditions` | array | 已知病史（如高血压、糖尿病） |
| `medications` | array | 当前用药清单 |
| `allergies` | array | 过敏史 |
| `updated_at` | datetime | 最近更新时间 |

### 3.3 `GET /health`
- 返回服务存活与依赖（Qdrant/DB/LLM）连通性，供 Railway 探活。

## 4. Agent 统一接口契约

所有子 Agent 实现同一调用协定，便于 Orchestrator 统一编排与替换。

**输入 `AgentInput`**
| 字段 | 类型 | 说明 |
|------|------|------|
| `query` | string | 经路由器/改写后的查询文本 |
| `profile` | object | 用户健康档案快照 |
| `history` | array | 最近 N 轮对话上下文 |
| `metadata` | object | 路由附加信息（疾病类型、子任务标签等） |

**输出 `AgentOutput`**
| 字段 | 类型 | 说明 |
|------|------|------|
| `agent` | enum | 产出该结果的 Agent 名 |
| `content` | string | 结构化结论 / 自然语言片段 |
| `severity` | enum\|null | 严重度（仅导诊 Agent 产出） |
| `sources` | array | 引用来源 |
| `confidence` | enum | high/medium/low，供聚合器决策 |

## 5. 编排层细化

### 5.1 安全网关（Safety Gate）
- **双重检测**：
  1. 关键词匹配（"剧烈胸痛""昏迷""左臂麻"等高危词库）
  2. LLM 快速分类（输出 `is_emergency: true/false`）
- 任一命中 → short-circuit：跳过意图分类与所有 Agent，直接返回应急就医提示模板
- 输出：`{ emergency: bool, matched_rule }`

### 5.2 意图分类器
六类意图及路由目标：

| 意图 | 路由目标 |
|------|----------|
| 症状描述 | 症状分析 Agent |
| 用药咨询 | 用药管理 Agent |
| 就医导诊 | 导诊 Agent |
| 多意图混合 | 路由器拆解 → 多 Agent |
| 档案管理 | 档案服务写入逻辑 |
| 高风险输入 | （已被安全网关拦截） |

### 5.3 路由器
- 单意图：直接构造 `AgentInput` 分发
- 多意图：拆解为子任务序列，按依赖关系串行或并行调度
  - 典型串行链：症状分析 → 导诊（导诊依赖症状结论）
- Query 改写：对指代模糊输入（"那这个药呢"）结合 history 补全后再分发

### 5.4 聚合器
- 收集各 Agent 的 `AgentOutput`
- **冲突消解原则**：结论矛盾时取更保守者（如导诊与症状冲突 → 优先建议就医）
- 严重度取各结果中的最高级别，🔴 触发界面置顶标记
- 合成最终 `reply`，附加 `sources` 与 `severity`

## 6. 服务层细化

### 6.1 RAG 检索服务
检索流水线（四阶段）：
1. **Metadata 过滤**：按疾病类型缩小候选集
2. **Hybrid Search**：稠密向量（BGE-M3）+ BM25 稀疏检索加权融合，召回 Top-20
3. **Reranker 重排**：BGE-Reranker 精排，取 Top-5
4. **Query 改写**：多轮场景下先补全上下文再检索

**接口**：`retrieve(query, disease_filter, top_k) → [chunk{text, source, score}]`

### 6.2 LLM 服务
- DeepSeek API 统一封装，提供 `classify`（快速分类）与 `generate`（生成）两类调用
- 集中管理 Prompt 模板、超时、重试、成本控制

### 6.3 档案服务
- 健康档案 CRUD
- 将档案渲染为 System Prompt 片段，供每轮对话固定注入
- 写入实时持久化，下一轮立即生效

### 6.4 安全检测服务
- 维护高危关键词库与应急提示模板
- 提供免责声明文案
- 被安全网关与最终响应包络调用

### 6.5 会话记忆服务
- `ConversationBufferWindowMemory`，保留最近 5 轮
- 按 `session_id` 隔离

## 7. 数据层模型

### 7.1 Qdrant 向量集合
| 字段 | 说明 |
|------|------|
| `id` | 文档块唯一标识 |
| `vector` | BGE-M3 稠密向量 |
| `text` | 切片正文 |
| `payload.disease` | 疾病类型（高血压/糖尿病/冠心病），用于 Metadata 过滤 |
| `payload.source` | 来源指南名称/页码 |

### 7.2 用户档案表
`user_id, conditions, medications, allergies, updated_at`

### 7.3 药品结构化表
`drug_id, name, ingredients, indications, contraindications, dosage, source`
- 用药 Agent 的禁忌检查基于 `contraindications` 字段逐一比对

## 8. 离线 Ingestion 管道

独立于在线服务，系统上线前预跑一次（数据更新时重跑）：

```
卫健委/药监局 PDF
   │ 解析（PDF → 文本）
   ▼
文本切片（按章节/语义分块）
   │
   ├─ 指南类 → BGE-M3 向量化 → 写入 Qdrant（附 disease metadata）
   └─ 药品说明书 → 结构化字段抽取 → 写入药品表
```

**职责边界**：管道只负责知识库构建，不参与请求响应；与在线链路通过 Qdrant / DB 解耦。

## 9. 在线请求时序

### 9.1 单意图（症状）
```
用户 → API → 安全网关(通过) → 意图分类(症状)
     → 路由器 → 症状Agent → RAG[过滤→Hybrid→Rerank]
     → 注入档案 → LLM生成 → 聚合器 → API → 用户
```

### 9.2 高风险 short-circuit
```
用户 → API → 安全网关(命中)
     → 跳过分类与所有Agent → 应急就医提示 → API → 用户
```

### 9.3 多意图（用药 + 导诊）
```
意图分类(多意图) → 路由器拆解
   → 症状/用药Agent ─┐
   → 导诊Agent      ─┴→ 聚合器(取最保守) → API → 用户
```

---

## 10. 安全边界设计（横切关注点）

贯穿编排层与接口层的四道防线：
1. **免责声明**：所有响应包络强制附加"仅供参考，不构成医疗建议"
2. **高风险 Short-circuit**：双重检测命中即停止一切医疗建议生成
3. **严重度置顶**：🔴 结果在前端强制置顶，不可被遮挡
4. **拒绝确诊**：措辞限定为"可能相关""建议咨询医生确认"，不做确诊性表述

## 11. 部署拓扑

```
Streamlit 前端 ──HTTP──> FastAPI 后端(Railway)
                              │
            ┌─────────────────┼──────────────────┐
            ▼                 ▼                  ▼
     Qdrant Cloud        SQLite/PG          DeepSeek API
     (向量检索)          (档案/药品)         (推理生成)
```

- 前后端分离，后端无状态（会话记忆按 session_id 隔离，可外置）
- 第三方依赖（Qdrant Cloud / DeepSeek）通过配置注入，便于替换
